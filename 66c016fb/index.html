JVM基础

运行时数据区结构

线程共享

  • 方法区：存放类的信息，常量信息， 字符串常量池信息等。Java8叫元空间，之前叫永久代。虽然Java虚拟机将方法区称为堆的一个逻辑部分，但它有个别名叫Non-Heap（非堆），目的是将它与Java的堆区分开。

  • 堆：用于存放对象的new出来的实例，几乎所有对象都在堆上分配内存。

线程私有

  • 程序计数器：用来存储指向下一条指令的地址，也即将要执行的指令代码

  • 栈：Java线程执行方法的内存模型，一个线程对应一个栈，每个方法在执行的同时都会创建一个栈帧。不存在垃圾回收的问题，在线程结束时自动释放。生命周期和线程一致。

  • 本地方法栈：本地方法栈为JVM执行native方法服务。

堆

  • ​ >=1.8
    • 新生代：类出生、成长、消亡的区域，一个类在这里产生，应用，后背垃圾回收器收集，结束生命。
      • eden：所有类都是在伊甸区new出来的
      • from survivor：当eden区的空间用完时，程序有需要创建对象，JMV垃圾回收器将eden区进行垃圾回收，将eden区中不再被其他对象应用的对象进行销毁（销毁无用对象），将剩余对象转移至From Survivor区。
      • to survivor：若From Survivor区也满了，则对该区进行垃圾回收，然后移动到To Survivor区。
    • 老年代：当(对象)或者(新生代经过多次GC仍存活的对象)移至老年区。若老年区也满了，这时将发生Major GC(Full GC)，进行老年区的内存清理。若老年区执行了Full GC后仍发现不能对对象进行保存，或抛出OOM的异常。
  • <1.8
    • 新生代(1/3)
    • 老年代(2/3)
  • GC
    • minor:新生代
    • major:老年代
    • full:整个堆

方法区

  • ​ >=1.8

    • Meta Space(元空间)
      • 在JDK1.8之后，元空间代替了永久代，是对JVM规范中方法区的实现，区别在于元数据区不在虚拟机中，而是用的本地内存，永久代在虚拟机中，永久代逻辑接口也属于堆，但物理上不属于。

  • <1.8

    • Perm(永久代)

JVM内存区域总结

JVM内存区域

线程共享

  • 方法区
    • 作用：即永代区，用于存放被JVM加载的类信息、常量、静态变量、即时编译器编译后的代码等数据，运行时常量池是方法区的一部分，用于存放编译器生成的各种字面量和符号引用。
    • 元空间：JDK1.8后元空间取代了永代区
  • 堆
    • 作用：创建的对象和数组都保存在java堆内存中，也是进行垃圾收集的最重要的内存区域。采用分代收集算法
    • 结构：
      • 新生代
        • Eden
        • From Survivor
        • To Survivor
      • 老年代

线程私有

  • 程序计数器：一块较小的内存空间, 是当前线程所执行的字节码的行号指示器，每条线程都要有一个独立的程序计数器，唯一一个没有OOM区域。正在执行 java 方法的话，计数器记录的是虚拟机字节码指令的地址（当前指令的地址）。如 果还是Native方法，则为空。
  • 虚拟机栈：是描述java方法执行的内存模型，每个方法在执行的同时都会创建一个栈帧（Stack Frame） 用于存储局部变量表、操作数栈、动态链接、方法出口等信息。每一个方法从调用直至执行完成 的过程，就对应着一个栈帧在虚拟机栈中入栈到出栈的过程。 不存在垃圾回收问题，只要线程一结束该栈就释放，生命周期和线程一致
  • 本地方法栈：和栈作用很相似，区别不过是Java栈为JVM执行Java方法服务，而本地方法栈为JVM执行native方法服务。登记native 方法，在Execution Engine执行时加载本地方法库。

垃圾收集器

简介：
  1. Java垃圾收集器是JVM的三个重要模块（+解释器，多线程机制）之一，为应用程序提供内存的自动分配、自动回收功能，这两个操作都在Java堆上。
  2. 某一个时点，一个对象如果有一个以上的Reference指向它，即表示存活，否则死亡，是为垃圾，可被回收。
  3. 垃圾回收需要消耗CPU、线程、时间等资源。所以回收不是实时发生，当内存消耗完或到达某一指标时，会触发垃圾回收操作。
回收类型：

  1. 串行回收器(Serial Collector)

    单线程执行回收操作，回收期间暂停所有应用线程的执行。

  2. 并行回收器

    并行回收器在串行基础上做了改进，它可以使用多个线程同时进行垃圾回收，对于计算能力强的计算机而言，可有效缩短垃圾回收所需的实际时间。

  3. 并行回收集器（Parallel GC）

  4. 并CMS(并发GC)收集器

  5. G1回收器

概念

内存溢出与内存泄露
概念
  1. 内存溢出：程序在申请内存时，没有足够的内存空间，出现OOM。例：申请了一个Integer，但存了Long才能存下的数据。
  2. 内存泄露：指程序申请内存后，无法释放已申请的内存空间，也就是指你向系统申请分配内存进行使用(new)，可是使用完了以后却不归还(delete)，结果你申请到的那块内存你自己也不能再访问。一次内存泄露的危害可以忽略，但内存泄露堆积后果很严重，无论多少内存，迟早会被占光。内存泄露会导致内存溢出。
发生的方式
  1. 常发性内存泄漏：发生内存泄漏的代码会被多次执行到，每次被执行的时候都会导致一块内存泄漏
  2. 偶发性内存泄漏：发生内存泄漏的代码只有在某些特定环境或操作过程下才会发生。常发性和偶发性是相对的。对于特定的环境，偶发性的也许就变成了常发性的。所以测试环境和测试方法对检测内存泄漏至关重要。
  3. 一次性内存泄漏：发生内存泄漏的代码只会被执行一次，或者由于算法上的缺陷，导致总会有一块仅且一块内存发生泄漏。比如，在类的构造函数中分配内存，在析构函数中却没有释放该内存，所以内存泄漏只会发生一次。
  4. 隐式内存泄漏：程序在运行过程中不停的分配内存，但是直到结束的时候才释放内存。严格的说这里并没有发生内存泄漏，因为最终程序释放了所有申请的内存。但是对于一个服务器程序，需要运行几天，几周甚至几个月，不及时释放内存也可能导致最终耗尽系统的所有内存。所以，我们称这类内存泄漏为隐式内存泄漏。
危害

从用户使用程序的角度看，内存泄漏不会产生什么危害，对于一般用户，不会感觉到泄漏的存在。但对于一个服务器程序，需要运行几天，几周甚至几个月，不及时释放内存也可能导致最终耗尽系统的所有内存。所以，我们称这类内存泄漏为隐式内存泄漏。
